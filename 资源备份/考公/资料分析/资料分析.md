##### Java使用 CompletableFuture方式创建多线程时如何指定多个异步任务之间的执行顺序？

> 使用 上一个异步任务. thenAccept、thenApply、thenCompose 方法
>
> ```
>   CompletableFuture<Void> voidCompletableFuture1= CompletableFuture.runAsync(() -> {
>             for (int i = 0; i < 100; i++) {
>                 System.out.println("我是线程1" + i);
>             }
>         });
> 
>         // 指定异步顺序，不带返回值
>         // CompletableFuture<Void> voidCompletableFuture2 = voidCompletableFuture1.thenAccept(state -> {
>         //     for (int i = 0; i < 100; i++) {
>         //         System.out.println("我是线程2" + i);
>         //     }
>         // });
>         // 指定异步顺序，带返回值
>         CompletableFuture<String> voidCompletableFuture2 = voidCompletableFuture1.thenApply(state -> {
>             for (int i = 0; i < 100; i++) {
>                 System.out.println("我是线程2" + i);
>             }
>             return "我是返回值";
>         });
>         // 使用thenCompose串行执行异步任务，可以将前一个任务的结果传递给后一个任务
>         CompletableFuture<String> composeFuture = voidCompletableFuture2.thenCompose(result -> CompletableFuture.supplyAsync(() -> {
>             System.out.println("我是thenCompose中的线程，获取到上一个任务的结果: " + result);
>             return "thenCompose的返回值";
>         }));
> 
>         // CompletableFuture<Void> voidCompletableFuture2=   CompletableFuture.runAsync(() -> {
>         //     for (int i = 0; i < 100; i++) {
>         //         System.out.println("我是线程2" + i);
>         //     }
>         // });
> 
>         // 组装异步结果，保证异步处理完毕
>         CompletableFuture.allOf(voidCompletableFuture1, voidCompletableFuture2).join();
>         System.out.println("我是异步处理完毕");
> 
>         try {
>          	// 使用get可以手动获取异步方法的返回值
>             String s = voidCompletableFuture2.get();
>             System.out.println(s);
>         } catch (InterruptedException e) {
>             throw new RuntimeException(e);
>         } catch (ExecutionException e) {
>             throw new RuntimeException(e);
>         }
> ```

##### 什么是分库分表?

> 将数据分散在多个数据库或表中，保证系统的扩展性、性能和可用性。

##### 为什么需要分库分表？

> 单机数据库在长时间使用时，会积累大量的数据和业务信息。此时数据库就是产生瓶颈，出现资源报警和慢查询问题。单机数据中能承载的IO和数据吞吐量是有限的，当并发数上来之后，数据库就顶不住了。如果数据表的数据量过大，也会导致B+树的层级过高从而导致查询缓慢。树的层级越高，IO次数越多，查询就越慢。

##### 分库分表有几种方式？

> 水平就是原结构不变，数据量扩展。垂直就是拆分原结构，按功能/字段/业务分离不常用的字段，提高可用性和维护性
>
> 针对于数据表：
>
> 水平分表(横向将一张表拆成多张)：将同一张表的数据，按行的方式分散到多张表中。比如：可以根据用户的id范围，将用户分成多张表（user_1,user_2,user_3）
>
> 垂直分表(纵向将一张表拆成多张)：将同一张表的数据，按列的方式分散到多张表中。减少每张表的字段数量，从而提高查询效率。
>
> 针对于数据库：
>
> 水平分库：将相同的表结构复制到另外的数据库中。每个数据库中的表结构是相同的。可以减少单一数据库的压力，在大数据量的时候可以提升读写的性能。
>
> 垂直分库：将数据分散在不同的数据库中，可以根据业务功能和模块进行分库，比如可以将用户数据和订单数据存储在不同的数据库中。

##### 分库分表的优点是什么？

> 提高性能：降低了数据库和数据表中的数据，减少了B+树的层级，减少了IO的次数，提高查询的性能。
>
> 可扩展性：可以增加新的数据库和数据表扩展业务
>
> 容错性：某一个数据库或数据表出问题不会影响到其他的数据库和数据表

##### 分库分表的缺点是什么？

> 数据的查询、一致性和事务变的复杂，提高的维护的成本。



分库分表之后数据库的单机事务就无法使用了，需要使用分布式事务。

##### TCP的三次握手是什么？

> 三次握手用于==建立连接==，发生在客户端和服务器之间。在传输层
>
> 第一次：客户端发送一个SYN(同步序列编号)消息给服务器
>
> 第二次：服务器收到后回复一个SYN-ACK(同步序列编号-确认)消息
>
> 第三次：客户端最后发生一个ACK(确认)消息用来确认已经收到了服务器端发送的SYN-ACK消息
>
> 从而完成三次握手，建立一个可靠的TCP连接
>

##### TCP的四次挥手是什么？

> 四次挥手用于==安全==的==关闭==之前==建立的连接==，发生在客户端和服务器之间。在传输层
>
> 第一次：客户端发送FIN包
>
> 第二次：服务器收到FIN包，回复ACK包
>
> 第三次：服务器==收到全部数据==后，发送FIN包
>
> 第四次：客户端发送最后一个ACK包

客户端（Client）              服务器（Server）
    |     FIN（1） --------->    |   ← 第1次挥手：客户端说“我要断了”
    |                            |
    |     <--------- ACK（2）    |   ← 第2次挥手：服务端说“收到，等我处理完”
    |                            |
    |     <--------- FIN（3）    |   ← 第3次挥手：服务端说“我也处理完了，要断了”
    |     ACK（4） --------->    |   ← 第4次挥手：客户端回应“收到，可以断了”
